StreamDB turns a [Durable&nbsp;Stream](/products/durable-streams) into a typed,
reactive database. You provide a [StandardSchema](https://standardschema.dev),
StreamDB gives you multiplexed, durable state with sync.

> [!Warning] ✨&nbsp; One-shot using TanStack Intent
> [Build and provision in one-shot](#getting-started) with your coding agent. Read the [StreamDB docs](https://durable-streams.com/stream-db).

Agent sessions need state. Tool calls, messages, active generations. User presence, agent registration. All the stuff in and around the agent loop.

This state should be real-time and reactive as well as persistent and forkable. In an ideal world you can just declare a schema and it all just works &mdash; typed, synced and reactive.

In practice you end up wiring together a database, a pub/sub layer, custom sync logic, and retry handling. Without type-safety across the pipe, you
write validation layers at every boundary &mdash; producer, consumer,
storage. Every project is a bespoke stack. Coordination is hand-rolled.
State breaks on disconnect. Collaboration is an afterthought.

## StreamDB

StreamDB wraps a Durable&nbsp;Stream with a StandardSchema to give you typed, reactive collections. You define your entity types &mdash; messages, presence, agents, whatever your session needs &mdash; as Standard Schema objects.

StreamDB routes events by type into [TanStack&nbsp;DB](https://tanstack.com/db) collections, with end-to-end type-safety from producer to consumer.

Multiplexing is built in. Multiple entity types coexist on a single stream, each routed to its own collection. The stream is the source of truth. Collections materialize state by applying events in order. Clients join from any offset and catch up.

<!-- ASSET: diagram showing StandardSchema → StreamDB → typed collections

### Reactive by default

Collections are TanStack&nbsp;DB collections &mdash; live queries with differential dataflow. When a new event arrives on the stream, only the affected data recalculates. Sub-millisecond reactivity.

Derived collections compose. You can filter messages, aggregate presence and [join up]() your session data with your existing app data.

### Durable by default

The underlying Durable&nbsp;Stream is persistent and addressable. State survives disconnects, refreshes, restarts. Clients resume from their last offset &mdash; no re-fetching, no state loss.

Everything is multi-user, multi-tab, multi-device and multi-agent out of the box. All reading and writing to the same stream.

### Define your schema

Start with Standard Schema objects for each entity type in your session.
Combine them into a unified schema with `createStateSchema` &mdash; this
maps each type to a collection with a primary key.

```ts
import { z } from 'zod'
import { createStateSchema } from '@durable-streams/state'

const schema = createStateSchema({
  messages: {
    schema: z.object({
      id: z.string(),
      role: z.enum(['user', 'assistant', 'system']),
      content: z.string(),
      createdAt: z.string(),
    }),
    type: 'message',
    primaryKey: 'id',
  },
  presence: {
    schema: z.object({
      userId: z.string(),
      status: z.enum(['online', 'offline']),
    }),
    type: 'presence',
    primaryKey: 'userId',
  },
  agents: {
    schema: z.object({
      agentId: z.string(),
      name: z.string(),
      endpoint: z.string(),
    }),
    type: 'agent',
    primaryKey: 'agentId',
  },
})
```

<!-- TONE: Matter of fact. "Here's the schema, here's what it does."
     No "as you can see" or "notice how". -->

Three entity types, one schema, one stream. Messages, presence, and agent
registration multiplexed together with full type-safety.

### Connect to a stream

`createStreamDB` connects your schema to a Durable&nbsp;Stream and gives
you a reactive, stream-backed database.

```ts
import { createStreamDB } from '@durable-streams/state'

const db = createStreamDB({
  streamOptions: {
    url: 'https://api.electric-sql.cloud/v1/streams/my-session',
    contentType: 'application/json',
  },
  state: schema,
})

await db.preload()
```

`preload()` reads from the beginning of the stream, materializes current
state, and stays connected for live updates. From this point,
`db.collections.messages`, `db.collections.presence`, and
`db.collections.agents` are live TanStack&nbsp;DB collections.

### Query reactively

Bind collections to your components with `useLiveQuery` &mdash; standard
TanStack&nbsp;DB.

```tsx
import { useLiveQuery } from '@tanstack/react-db'

function MessageList() {
  const { data: messages } = useLiveQuery((q) =>
    q
      .from({ msg: db.collections.messages })
      .orderBy(({ msg }) => msg.createdAt, 'asc')
  )

  return <List items={messages} />
}
```

Queries update incrementally via differential dataflow &mdash; no
re-scanning, no re-rendering unaffected rows. Derived views compose
naturally: filter, join, aggregate across collections.

```tsx
function OnlineAgents() {
  const { data: agents } = useLiveQuery((q) =>
    q.from({ agent: db.collections.agents })
  )

  return <AgentList items={agents} />
}
```

### Derive collections

<!-- STRUCTURAL: Show that StreamDB collections compose into derived views.
     The chunks → messages materialization is the killer example — raw
     stream data becomes structured UI state via live query pipelines. -->

Raw stream data often needs materializing into higher-level structures
&mdash; token chunks into complete messages, for example. Derived
collections let you do this declaratively, using live query pipelines
that update incrementally.

```tsx
import { createLiveQueryCollection, collect, count, minStr } from '@tanstack/db'

const messagesCollection = createLiveQueryCollection({
  query: (q) => {
    // Group chunks by messageId, collecting the raw rows
    const collected = q
      .from({ chunk: db.collections.chunks })
      .groupBy(({ chunk }) => chunk.messageId)
      .select(({ chunk }) => ({
        messageId: chunk.messageId,
        rows: collect(chunk),
        startedAt: minStr(chunk.createdAt),
        rowCount: count(chunk),
      }))

    // Materialize grouped chunks into messages
    return q
      .from({ collected })
      .orderBy(({ collected }) => collected.startedAt, 'asc')
      .fn.select(({ collected }) => materializeMessage(collected.rows))
  },
  getKey: (row) => row.id,
})
```

<!-- ASSET: link to materializeMessage source or explain briefly what it does -->

No for loops over client data. Differential dataflow means only changed
chunks trigger recalculation. Derived collections are themselves
TanStack&nbsp;DB collections &mdash; you can query, filter, and derive
further from them.

```tsx
// Derive pending tool call approvals from the materialized messages
const approvalsCollection = createLiveQueryCollection({
  query: (q) =>
    q
      .from({ message: messagesCollection })
      .fn.where(({ message }) =>
        message.parts.some(
          (p) => p.type === 'tool-call' &&
            p.approval?.needsApproval === true &&
            p.approval.approved === undefined
        )
      ),
  getKey: (row) => row.id,
})
```

Chunks sync on the stream. Messages materialize from chunks. Approvals
derive from messages. Each layer is reactive, typed, and incremental.

### Write with optimistic mutations

Writes go through TanStack&nbsp;DB's optimistic action system. Local state
updates instantly, persistence happens async.

```ts
const db = createStreamDB({
  streamOptions: { url: streamUrl, contentType: 'application/json' },
  state: schema,
  actions: ({ db, stream }) => ({
    sendMessage: {
      onMutate: (msg) => {
        db.collections.messages.insert(msg)
      },
      mutationFn: async (msg) => {
        const txid = crypto.randomUUID()
        await stream.append(
          schema.messages.insert({ value: msg, headers: { txid } })
        )
        await db.utils.awaitTxId(txid)
      },
    },
  }),
})

await db.actions.sendMessage({
  id: crypto.randomUUID(),
  role: 'user',
  content: 'Hello',
  createdAt: new Date().toISOString(),
})
```

<!-- TONE: The code is the pitch. Local insert is instant. Stream append
     is durable. awaitTxId confirms sync. That's the whole write path. -->

`onMutate` inserts into the local collection immediately &mdash; the UI
updates before the network round-trip. `mutationFn` appends to the
Durable&nbsp;Stream with a transaction ID, then waits for it to sync back.
If the write fails, TanStack&nbsp;DB rolls back the optimistic state
automatically.

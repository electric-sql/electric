Principals in Electric Agents

RFC summary for the team — #4306 · builds on #4305

Core Decisions

Required Every request needs a principal

All requests must include an Electric-Principal header identifying who is acting. Format: type:id (e.g. user:kyle, agent:ci-bot, service:deploy-hook). In dev mode, defaults to dev:local when absent.

Integrator Auth is not our problem

Electric Agents doesn't authenticate users or validate identity. The integrator's edge (reverse proxy, Clerk, Better Auth, SSO) authenticates the user and sets the header. We trust the header. We don't care what the identity is, just that there is one.

Integrator Authorization lives in handlers & tools

No framework-level permission model. The handler decides what tools/functions to give each agent based on ctx.principal. Handlers can load auth rules from any external API (Clerk, your own DB, a permissions service).

Electric Cloud API tokens for system principals

Electric Cloud needs to issue API tokens for non-human principals (system:cron, service:ci) that spawn agents, send messages, or trigger workflows. These tokens map to a specific principal identity in the header.

Required Agents are owned by principals

Every agent records created_by at spawn. Enables: querying all agents for a user, audit trails, cascading cleanup. Without this you can't build basics like "show me my agent sessions."

                    │  Integrator's Edge                │

          │  Can call Clerk API, query DB, etc.              │
          │  Decides what the agent can do                   │
          └─────────────────────────────────────────────┘
Handler Example
defineEntity("support-agent", {
  async handler(ctx) {
    const { type, id } = ctx.principal;  // { type: "user", id: "kyle" }

    // Build tools based on who the principal is
    const tools = [...ctx.electricTools];

    if (type === "service") {
      // CI/webhooks get a narrow tool set
      tools.push(statusTool);
    } else {
      // For users, fetch capabilities from your auth system
      const perms = await myAuthAPI.getPermissions(id);
      if (perms.canSearch) tools.push(searchTool);
      if (perms.isAdmin)  tools.push(adminTool);
    }

    ctx.useAgent({
      systemPrompt: "You are a support agent...",
      model: "claude-sonnet-4-20250514",
      tools,
    });
    await ctx.agent.run();
  }
});
Use Cases This Covers
Personal system

Solo developer

Dev mode defaults to dev:local. Zero auth config. Principals exist for observability but impose no restrictions.

Software factory

Multi-engineer team

Each engineer is a principal. Agents are scoped by what tools the handler gives them. CI bots get service:ci identity.

Customer app

SaaS with end users

Clerk/SSO authenticates users at the edge, sets the header. Handler loads per-customer permissions from your DB.

Known Future Layers

Not now, but the principal substrate supports these later

  • Public sharing links — HMAC-signed URLs for scoped read-only access to agent sessions, dashboards
  • Capability expressions — runtime-evaluated authorization conditions stored in principal entity streams (time, relationship, context-based)
  • Named capability sets — viewer / editor / admin as shortcuts over capability expressions
  • Delegation attenuation — child agents get provably narrower authority than parent
  • Cross-principal policy — framework-level rules for inter-principal messaging

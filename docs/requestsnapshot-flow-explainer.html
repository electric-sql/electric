Context

ShapeStream receives batches from long polling or SSE, runs them through #onMessages, and publishes them to all stream subscribers. Shape.requestSnapshot() depends on the injected snapshot batch being processed before the call resolves, because callers may read materialized rows immediately after awaiting it.

Server batch Long poll body or SSE messages ending at up-to-date.
#onMessages Updates stream state, filters duplicate snapshot rows.
#publish Calls every subscriber callback for the batch.
Shape Maintains materialized rows and notifies its subscribers.
User code May await requestSnapshot and then read currentRows.

Fix Options

The core choice is whether snapshot delivery is allowed to be out-of-band. If yes, make that explicit and narrow. If no, the publishing model needs a stronger scheduler than a single promise chain.

Option How it works Pros Risks
Current PR
Global reentrant bypass		 If #publish sees #isPublishing, it calls subscriber callbacks immediately instead of appending to #messageChain. Small patch. Fixes the observed self-deadlock. Bystander subscribers can be re-entered. SSE batches can bypass ordering whenever a publish is in flight.
Explicit escape hatch
Pass publish mode		 Change internals to something like #onMessages(batch, { publish: "reentrant-snapshot" }) and only allow bypass for snapshot injection. Makes semantics reviewable. Prevents accidental bypasses from normal SSE or long-poll batches. Still permits bystander subscriber reentrancy for snapshots unless paired with per-subscriber ordering.
Recommended
Per-subscriber queues		 Keep global batch ordering for normal messages, but deliver a reentrant snapshot through each subscriber's own queue. The subscriber that requested the snapshot can be handled specially or the request can await the snapshot's effects without re-entering other subscribers. Avoids the deadlock while preserving the important invariant that one subscriber is not called again before its prior callback completes. More code. Needs careful tests for unsubscribe, thrown callback errors, and snapshot control messages.
API split
Fetch vs inject		 Keep fetchSnapshot() pure, and make requestSnapshot() document a two-phase contract: update internal shape state synchronously, then notify subscribers on the normal chain. Clearer user-facing behavior. Can make immediate reads after await reliable without recursively publishing. May require moving responsibility from ShapeStream into Shape or another materialization layer.

Tests To Add

Missing

Bystander subscriber non-reentrancy

Subscriber A awaits requestSnapshot(). Subscriber B awaits unrelated work in its M1 callback. Assert B does not receive snapshot before its M1 callback returns, or explicitly document that this is allowed.

Missing

SSE batch ordering

Simulate two SSE up-to-date-delimited batches while the first publish is still awaiting. Assert M2 does not bypass M1 unless the new contract intentionally allows it.

Covered

Snapshot await consistency

The PR adds a unit test proving requestSnapshot() waits until the injected snapshot batch reaches subscribers before resolving.

Covered

Self-deadlock regression

The PR adds a unit test proving a subscriber can await requestSnapshot() without hanging.

Recommendation

Do not merge the current implementation as a hidden global bypass. Either narrow and document the out-of-band snapshot path, or move to a scheduler that preserves per-subscriber non-reentrancy while still allowing the subscriber that requested a snapshot to make progress.
